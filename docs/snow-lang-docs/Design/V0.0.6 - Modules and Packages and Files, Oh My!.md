# Details and Design
## Compilation
- The compiler is provided the root file, the file which contains the package statement
- Read this file and gather the package statement
- Execute the package statement and configure the package
- Compile dependencies (same way as this, no circular dependencies)
- Open the root file again this time skipping the package statement, compile it
	- For each import, search out and compile those files
	- This compilation is stage 1, does tokenization and syntax analysis. This loads enough data for the types to be loaded. Function implementation is simply cached as an offset into the file and it's length.
- When all files have been stage 1 compiled, output a file of the package name `.t` which contains the type information. This can be used later in compilations to not have to reload the type data
- When all files have been stage 1 compiled, do stage 2 compilation
	- Each file opens again only looking for the function implementations and compile those with the types we have made
	- This stage is full compilation with optimizations and all depending on input
	- This outputs `.o` files for the file's byte code.
- And finally an executable is created by linking all the `.o` files together
- For now the package statement only accepts `inter` for interpret, this will eventually be deprecated
## Package statement
- This statement is the build system of snow
- `package inter NAME;` is a default build which compiles only this app. The app name is NAME
- `package inter NAME {}` is the more complex build system which allows for dependencies and more. The inside of the scope is executed in the restricted interpreter (only `inter` for now)
	- Here you can set dependencies the output name and more
	- You set these on the object that is your package's name which is created for the scope
- Only 1 package statement can exist per package. Other package statements are added by dependencies of a package
```
package exe A {
	A.depends += package("../B/B.snw");
	let plat = CArgs().find("platform");
	if (plat && plat == "WIN32") {
		A.startFn = win32_main;
	}
	else {
		A.startFn = main;
	}
}
```
- The package object contains
	- `startFn`: The function that will be called to start this program. Defaults to `main`
	- `depends`: An array of packages that this object depends on (aka should build first)
		- By build first, this means gather all types for both projects, then we can build dependencies fully and build ourselves fully outside of each other. Then link the two together into a single object.
		- To construct a package use the `package` function providing arguments and a file to load
	- `coBuild`: Similar to depends, but isnt a dependent package, just also produce this package
	- `cargs`: The arguments to compile this with, defaults to the CArgs object unless set
	- `preBuildTasks`: An array of strings which represent command line arguments to run before building this package
	- `postBuildTasks`: An array of strings which represent command line arguments to run after building this package
	- 

## File Structure
- Package statements can only exist once in a folder hierarchy
	- Example
		- package A
			- 1
				- 2
					- package D // not allowed
		- package B // safe
			- package C // not allowed
- importing from outside your package's hierarchy requires that the package takes a dependency on the other package
- Libraries and Dynamic Libraries are allowed to have dependencies on each other
	- Executables and Restricted Interpreters cannot have dependencies taken on them
- Projects
	- A project is a collection of packages

## Modules
- A module is a logical block of code for organizational purposes
- A module is NOT a file, however a file is a module
- A file is a module of the name of the file
	- Everything inside the file is considered a part of the module
### Import Statements
- Import statements are how you include other modules to use
- Import statements are not defined by file structure, however the file structure is a part of the import flow
- `import some.folder.A;`
	- This statement could mean there is a file `PACKAGE_ROOT/some/folder/A.snw` or it could be a module defined in a file
	```
	// file some.snw
	module folder {
		module A {
		}
	}
	```
- When importing a module, the module name is used to then access the elements in said module:
	```
	// file some/folder/A.snw
	pub fn B() {}
	
	// file C.snw
	import some.folder.A;
	A.B();
	```
- If a module only contains one public or exported symbol, the module name can be used to reference that object (if it also has the same name)
	```
	// file A.snw
	pub class A {};
	
	// file B.snw
	pub fn B() {};
	
	// file C.snw
	import A;
	import B;
	let a = A();
	B();
	```
- Everything in a module is private by default, use the `pub` keyword to make that object callable from outside the module
	```
	// file A.snw
	pub class One {};
	class Two {};
	
	// file B.snw
	import A;
	let a = One();
	let b = Two(); // error, type Two is private
	```
- The `export` keyword is used to make something in a module both public to the module and public to the package. External packages can only use objects that are `export` from the another package. Using both `export` and `pub` is not required, but is not an error. 
- When importing a module to another module, they are privately imported
	```
	module A { pub class _A {} }
	module B {
		import A;
	}
	module C {
		import B;
		let x = B.A._A(); // error, A is not a public object in module B
	}
	```
	- You can however publicly import modules which does make them accessible. This means you can define shortcuts in your file hierarchy or group things together in a more convenient way than importing a ton of files
	```
	module A { pub class _A {} }
	module B {
		pub import A;
	}
	module C {
		import B;
		let x = B.A._A(); // safe
	}
	```
	- Similarly, you can export a module to make your external API have a similar usage
- You can rename modules to be easier to use in your code, or import two modules of the same name as different identifiers
	```
	// file A/Sub.snw
	pub fn _A() {}
	
	// file B/Sub.snw
	pub fn _B() {}
	
	// file C.snw
	import A.Sub;
	//import B.Sub; // error, the identifier Sub is already defined in this module
	import BSub as B.Sub; // safe
	
	Sub._A(); // A.Sub
	BSub._B(); // B.Sub
	```
	- In the case where a module has only one exported type, renaming a module import will NOT rename that type. The module is then required to access the type 
	- You can rename `pub` and `export` import statements to provide a different module name for the API
- Modules can be friends with one another. This allows a module to access privately defined objects in the friended module
	```
	module A {
		friend module B;
		class _A {};
	}
	module B {
		fn _B() { let x = _A(); } // safe since friends
	}
	```
	- Friends can only span within the package, not to other packages
- File imports are shared with all modules in the file, however a sub-module can import things and that is not shared with other modules in the file or the file's module
	```
	import std.io;
	module A {
		import std.map;
		// canuse std.io as if it imported it
	}
	module B {
		// cannot use std.map, it wasnt imported to the file or this module
	}
	// cannot use std.map, it wasnt imported to the file
	```
