# Details and Design
## Type Inspection
### Is Keyword
- This keyword allows you to compare that the type of a variable or generic is another type
- `T is SomeClass`, `T is K` (both generic types)
- Return is a compiler constant true or false, usable in where statements
### Has Keyword
- This keyword allows you to ask if the type has a function name or variable name
- The return type is a compiler constant optional type description
- `T has funcName`
- Can only `has` for a public thing on the object
### Typeof Keyword
- This keyword returns a constant type description object from a value
- `typeof(T).name()`, `typeof(someVariable).name()`
### Type Description Object (`__typedesc`)
- This object does not formally exist in anyway that is instantiable by users. The compiler generates this object when a type description is needed
- The type description object has the following API, all functions and variables are compiler constants, so they can be used in where statements and are evaluated at compile time
	- `fn isBuiltIn(): bool`: Returns true if the type is a compiler built-in type
	- `fn isNumber(): bool`: Returns true if the type if an integral or floating point number
	- `fn isFunction(): bool`: Returns true if the type is a function
	- `fn isTuple(): bool`: Returns true if the type is a tuple type
	- `fn isArray(): bool`: Returns true if the type is an array type
	- `fn isMap(): bool`: Returns true if the type is a map type
	- `fn isOptional(): bool`
	- `fn isImmutable(): bool`
	- `fn isGeneric(): bool`
	- `fn getUnderlyingTypes(): [__typedesc]`: Returns the tuple types, array type, function argument types, or map key and value types, can be empty if not one of those types
	- `fn getReturnType(): __typedesc?`: Returns the return type of a function or empty if not a function
## Function
- Default Values
	- `fn func(x : u32 = 0);`
	- Default values can be anywhere in the parameter list
	- `fn func(x : u32 = 0, y : u32);`
		- `func(4); // error, parameter 1 ('y') was not provided a value`
		- `func(y = 3); // allowed, x is 0, y is 3`
		- `func(3, 4); // allowed`
		- `func(x = 3, y = 4); // allowed`
- Variadics
	- Represented as an argument type `...`, this is a named parameter pack in the function signature. It is defined as a tuple of the arguments. The variadic argument must be the last parameter in the function parameter list.
	```
	fn func(args : ...) {}
	func(4, 3, 2, 1); // func provided with a tuple of 4 values, all i32 with the values 4, 3, 2, and 1
	func(); // args is empty tuple ()
	
	fn func2(a : u32, args : ...) {}
	func2(4, 3, 2, 1); // a = 4, then args is a tuple (3, 2, 1)
	
	fn func3(a : u32, b : u32 = 4, ...) {}
	func3(4, 3, 2, 1); // a = 4, b = 3, args = (2, 1)
	func3(a = 4, 3, 2, 1); // error positional arguments cannot be placed after default value args
	func3(a = 4, args = (3, 2, 1)); // a = 4, b = 4, args = (3, 2, 1)
	```
	- What about chaining variadics where one function which takes a variadic wishes to pass its variadic object to another function taking a variadic? The same way you unwrap tuples!
	```
	fn A(args : ...) {}
	fn B(args : ...) {
		A(args...); // Here the `...` passes the variadic as individual elements
		// dont want to pass all the elements, make a new tuple with all values you want to pass, then `...` that tuple
	}
	```
## Switch
- A switch statement is essentially a bunch of if statements wrapped together
```
switch (someValue) {
	Expression0: SingleStatement;
	Expression1: {
		MultiStatement0;
		MultiStatement1;
	}
	_: DefaultStatement;
}
```
- Default must be the last element in the switch statement if used
- Default does not have to be used if the other cases exhaust all possible values
- `_:` defines the default case
	- To do nothing with a default case, use `_: break;` or `_:{}`
- Each case is just a boolean evaluation and can be fairly complex
	- Due to this, the compiler classifies case statements as simple or complex. The compiler can only evaluate case value exhaustion for simple statements
	- Simple statements: `x > 10`, `12`, `3 || 4`, `1..4`, `y >= 4 && y <= 5`, `!10`
	- Complex statements: `isFoo(x)`, `((x - 10) * thinger() / foo) > 1`
	```
	switch (x) {
		x > 10: print("greater");
		x < 10: print("less");
		10: print("same");
	}
	```
- `fallthrough` is a unique keyword that only works in switch statements, it is a single word statement similar to `return;`
	- This lets more than one case statement be executed
	```
	switch (v) {
		0: { print("hello"); fallthrough; }
		1: print("world");
		_: print("Unknown");
	}
	```
	- In this example, passing `0` will print both `hello` and `world`. Passing `1` will only print `world`, and passing any other value will print `unknown`
- `break` can also be used in switch statements which acts as a return from the switch statement, but not the function
	```
	switch (v) {
		0 || 1 || 2: {
			print("hello");
			if (0) { break; }
			if (1) { return; }
			print("world");
		}
		_: print("Unknown");
	}
	print("after");
	```
	- In this example, passing `0` prints `hello` and `after`; passing `1` prints `hello`, `world`, and `after`; and passing 2 will print only `hello`. The return statement leaves the function, but the break statement only leaves the switch statement
- Switch statements can also work on types using `where switch`
	```
	generic<T> get10() -> T {
		where switch(T) {
			string: return "10";
			f32: fallthrough;
			f64: return 10.0;
			T.isIntegral(): return 10;
			_: panic("Unknown Type");
		}
	}
	```
	- Where switch must always contain a default as any type could be passed in
- Switch statements can return a value from the statement similar to if statements
	- Using `use` will return a value from the switch statement
	- When a switch statement uses `use`, all code paths must return a value
	```
	let x = switch(y) {
		0: use 5;
		1: {
			// some calc
			use calc;
		}
		_: use undefined;
	}
	```
## Generics
- Generics are a way to provide a type to another object which fully realizes that objects type and perform generic operations over the nebulous type
- Where clauses allow you to restrict what kinds of types can be used in a generic. If you want numbers but not floating point numbers, you can restrict a function to only accept integrals with a where clause.
	- `where(T is i32)`
	- `where { ...; use val;}`
	- `generic<T> where(T is i32 || T is u32) fn is32BitNum() {}; // produces error when non-32 bit integer is passed in`
- Generics use the `generic` keyword and are followed by the names for the types that will be used
	- After the generic statement, you can optionally use the `where` clause
- Generic declarations come before the thing they are making generic, such as a class or function
	```
	generic<T> class Name {}
	generic<T> fn Name() {}
	```
- Generic usages are done after the type name
	```
	generic<T> class GenClass {}
	// usage:
	let x = GenClass<i32>();
	
	generic<T> fn GenFn() {}
	// usage:
	let x = GenFn<i32>();
	```
	- Functions are capable of automatically detecting the type based on the arguments provided
	- However, all usages of the type must be valid to use, in these cases uses of where clauses help show your intent
		```
		// This requires that T both has bar and baz functions
		generic<T> fn foo(t : T) {
			t.bar();
			t.baz();
		}
		
		class BarOnly { pub fn bar() {} };
		let one = BarOnly();
		foo(one); // error
		```
	- Classes are also capable of this for their member functions AND their initializers
	- In both cases though, if a type isnt filled out by parameters (such as one not being in the parameter list), it must still be set by the caller
- Using generics follow the same rules as function parameters including variadics. In the case of variadics, they essentially create a tuple of types directly rather than a tuple of values. However there is one difference in the generics declaration than function declarations for variadics:
	```
	generic<T, U, K, L...> // the variadic object is named L
	generic<T...> // the variadic object is named T
	generic<T, ...> // error, variadic unnamed
	```
- Where statements can also be used outside of generics as a compilation trigger or inside functions (generics) to do different things based on the type
	- NOTE: Im not sure I like that where statements can be used as compiler guards, might want to make that a stretch goal or come back to it with a different design
	```
	// NOTE might not be doing this out of function
	where(std.compiler.platform == std.compiler.win32) {
	import win32;
	} otherwise {
	import linux;
	}
	```
	- The otherwise statement is simile to else in if-else expressions
	```
	generic<T> fn foo(): T {
		where (T is i32) {
			return 22;
		}
		otherwise where(T is f32) {
			return 10.0f;
		}
		otherwise {
			panic("unknown");
		}
	}
	```
